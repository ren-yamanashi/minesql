## 外部キー

- ON DELETE / ON UPDATE のアクションは RESTRICT のみサポートする
  - 参照先のレコードが参照元から参照されている場合、削除や更新を拒否する
- 自己参照外部キー (テーブルが自分自身を参照する外部キー) はサポートしない
- 外部キーカラム (子テーブル側) にはインデックス (KEY, UNIQUE KEY, PRIMARY KEY のいずれか) が必須
  - 自動作成はされず、明示的にインデックスを作成する必要がある (作成しないとエラーになる)
- 外部キーの参照先カラム (制約によって参照されるカラム) には PRIMARY KEY または UNIQUE KEY が必須
  - 参照先のカラムが一意であることを保証しないと、参照元レコードの更新時に不都合が出る

  - 参照先のカラムが一意でないと以下のような問題が発生する
    - 以下のようなテーブルがあると仮定

      ```sql
      -- もし非ユニークな参照先を許した場合
      users: | id | name  |
             | 1  | Alice |
             | 2  | Alice |
      orders: | id | user_name |  -- REFERENCES users(name)
              | 1  | Alice     |
      ```

      - `users.name` が非ユニークなため、`orders.user_name` の値 `Alice` がどちらの `users` レコードを参照しているのか特定できない
      - そのため、仮に `users` テーブルの `id=1` のカラムを削除した際に、`orders` テーブルの `id=1` のレコードがあるから RESTRICT で削除を拒否したら良いのか、それとも `users` テーブルの `id=2` のカラムがあるから削除可能として良いのか判断できない
