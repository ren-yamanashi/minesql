# Durability を実現するためのアルゴリズム

- バッファプール (メモリ) がいっぱいになった時に LRU ポリシーでページを選んでディスクに書き戻す
- つまり、バッファプールに載っているページがいつディスクに書き込まれるかは、バッファプールの都合で決まり、トランザクションとは無関係
- そのため以下のような問題が発生する可能性がある
  - COMMIT したのに消える: COMMIT 後にクラッシュして、まだディスクに書き戻されていないページが消える
  - ROLLBACK したのに残る: ROLLBACK 後にクラッシュして、まだディスクに書き戻されていないページが残る
- COMMIT した後に Flush (ディスクに書き戻す) したとしても、以下のようなケースで問題が起きる
  - $T_1$ が COMMIT したタイミングで Flush すると、$T_2$ のまだコミットしていない変更も一緒にディスクに書き込まれてしまい、$T_2$ が ROLLBACK した後にクラッシュすると、$T_2$ の変更が残ってしまう
- これらの問題を解決するために、Durability (永続性) を実現するためのアルゴリズムが必要になる

## Write-Ahead Logging (WAL)

- トランザクションの変更内容を、実際のデータページに書き込む前に、ログ (WAL) に先に書き込む方式
- ログには、以下の内容が記録される
  - どのトランザクションによる変更か
  - どのデータアイテムに対する変更か
  - どのように変更したか
- MySQL InnoDB で言うところの InnoDB Log (Redo Log) がこれに該当する
- WAL を使用することで、以下のパフォーマンスを重視した挙動を実現することができる
  - NO-FORCE: COMMIT 時にページを書き出さなくて良い (対義語は FORCE)
    - WAL を見れば、COMMIT したトランザクションの変更内容がわかるため、ページに書き戻されていなくても復旧 (REDO) できる
  - STEAL: 未コミットのトランザクションが変更したページでも、バッファが足りなければディスクに書きだせる (対義語は NO-STEAL)
    - WAL を見れば、未コミットのトランザクションの変更内容もわかるため、復旧 (UNDO) できる

## Log Sequence Number (LSN)

- 各 WAL レコードには LSN と言う連番が付与される
- LSN 配下の役割を果たす
  - 順序の保証: WAL レコードの時系列順序を保証
  - Page LSN: 各ページは、「このページに最後に適用された WAL の LSN」 を記録する
    - リカバリ時の冪等性を保証するために使用
    - リカバリ時、Page LSN をみて、「このページには LSN=100 まで適用済み」と分かれば、LSN=100 以下の WAL レコードの適用はスキップできる

## ARIES

- WAL にはデータベースのすべての変更履歴が記録されている
- ARIES (Algorithm for Recovery and Isolation Exploiting Semantics) を使用して、WAL をもとにクラッシュリカバリを行う

ARIES は以下の 3 ステップでクラッシュリカバリを行う

### 1. Analysis (分析フェーズ)

- WAL をある位置から最後までスキャンし、以下を特定する
  - コミット済みのトランザクション
  - 未コミットのトランザクション (`BEGIN` があるが `COMMIT`, `ROLLBACK` がないもの)

### 2. REDO (再実行フェーズ)

- WAL を走査し、コミット済みかどうかに関わらずすべての操作の再実行を試みる
- この際に、Page LSN を参照し、すでに適用済みの操作はスキップする
- このフェーズで、クラッシュ直前の状態が復元される

### 3. UNDO (取り消しフェーズ)

- 未コミットのトランザクションの操作を Undo ログを使用して逆順に取り消す
- この際に CLR (Compensation Log Record) を書くことで、Undo 自体もリカバリ可能にする

## Fuzzy Checkpoint

- ARIES の仕組みの一部で、システムを停止せずに Checkpoint を取る方式
  - https://dl.acm.org/doi/10.1145/128765.128770
- 定期的にデータベースの状態 (アクティブなトランザクション、ダーティーページなど) を記録し、Analysis フェーズの開始位置を決定するもの
- Checkpoint があれば、Checkpoint 時点でアクティブだったトランザクションの最も古い LSN より前の WAL は不要になるため、WAL を切り詰めることができる
- Fuzzy Checkpoint では、ダーティーページの Flush は行わず、リカバリに必要な以下の情報だけを記録する (以下の情報のスナップショットを取得して WAL に書き込む)
  - ATT (Active Transaction Table): 現在アクティブなトランザクション一覧と、各トランザクションの最後の LSN
  - DPT (Dirty Page Table): 現在のダーティーページ一覧と、各ページが最初にダーティーになった時の LSN (RecLSN)
    - DPT 内の最小の RecLSN より前の変更は、必ずディスクに反映されている
