# トランザクション

## 概要

- トランザクションは、一連の読み書き操作をまとめたもの
- ACID でいうところの Atomicity (原子性), Isolation (独立性), Durability (永続性) の特性を保証すべき
  - Consistency (一貫性) つまりビジネスロジックの整合性を保証する性質については、DBMS ではなくアプリケーション側の責務 (例えば口座残高がマイナスにならない。ユーザーIDは重複しない。など)
    - 「データ思考アプリケーションデザイン」でも以下のような記述があったり  
    > ACIDはほとんどマーケティング用語  
      特にConcistencyはACIDに含まれない

### Atomicity (原子性)

- すべて成功するか、すべて失敗するかを保証する性質
- トランザクションが完了できなかった場合、トランザクション開始前の状態に戻す必要がある
- トランザクション中に行った変更操作を記録しておき、失敗した場合にはそれらを逆順に実行して戻すことで実現する

※ Atomicity は単一のトランザクション内で完結する性質

### Isolation (独立性)

- 複数のトランザクションが同時に実行される状況において、互いに干渉しないことを保証する性質
- DBMS 側で同時に実行されているトランザクションが常に 1 つになるように他のトランザクションを待たせておけば実現できるが、スループットが出ず現実的ではない
- そのため通常は複数のトランザクションを並行 (Concurrent) に実行し、各トランザクションの個々の操作は時系列的に入り混じる

### Durability (永続性)

- 一度 Commit されたトランザクションによる変更は、後に障害などが発生しても失われないことを保証する性質
  - 不揮発性ストレージ (e.g. SSD, HDD) に記録された時点で保証されると考える
- Commit するたびにディスクに書き込めば実現できるが、それだと遅すぎて現実的ではないので、実際には揮発性メモリ (バッファプール) 上でデータの読み書きを行う
- 揮発性メモリを使うトレードオフとして、復旧のためのアルゴリズムが要求される
  - MySQL だとクラッシュリカバリという仕組みがある
