# オプティマイザ統計情報の収集方法

## Motivation

Planner にコストベースの最適化機能を追加するにあたり、テーブルやインデックスの統計情報をどのように収集・管理するかを決める必要があった

## Decisions

閾値ベース ANALYZE でテーブルやインデックスの統計情報を収集する

## Context

### 前提

- コストベースの最適化機能の追加時点では、JOIN をサポートしていなかった
- コスト算出に際して必要な情報はざっくり以下の通り

| 値 | 意味 | 用途 |
| --- | --- | --- |
| R(T) | テーブル T のレコード数 | すべてのコスト計算で必要 |
| B(T) | テーブルの総 (リーフ) ページ数 (リーフノードの数) | テーブルスキャンのコスト計算で必要 |
| V(T, F) | テーブル T のフィールド F の異なる値の数 (重複を除いた値の数) | 選択スキャンのコスト計算で必要 |
| H | セカンダリインデックスの B+Tree の高さ (ルートからリーフまでのページ数) | インデックススキャンのコスト計算で必要 |
| min(F), max(F) | カラム F の最小値・最大値 | 範囲クエリのコスト計算で必要 (min/max が不明な場合は固定の推定値を使う) |

<br />

この統計情報の算出方法として以下の二つの方法を考えた

### 1. DML 実行の度に更新する

INSERT/DELETE/UPDATE などの DML 実行時に、上述したすべての統計情報を更新する方法 (例: R(T) なら INSERT で +1、DELETE で -1)

### 2. 閾値ベース ANALYZE で収集する

INSERT/DELETE/UPDATE などの DML 実行時に "変更行数" のみを記録 (カウント) しておき、一定の閾値を超えたタイミングで、次のクエリ最適化時に統計情報を計算する方法

より具体的には、以下の値をテーブルごとに保持しておく

- last_analyze_row_count: 前回 ANALYZE 時に算出した R(T)
- dirty_count: 前回 ANALYZE 以降の変更行数 (INSERT/DELETE/UPDATE ごとに +1)

そして以下のような閾値を決め、その値を超えたタイミングで、次のクエリ最適化時に統計情報を計算する

- dirty_count > last_analyze_row_count * 0.1  (10% 以上変更された)

### 判断基準

上記の二つの案のどちらを採用するかの決定に際し、以下の観点で判断した

- 実装コスト
- 将来的な拡張性
  - 例えば JOIN 実装時にどれくらいの手戻りが発生するかなど
- 機能の必要性の範囲

### 判断結果

#### 実装コスト

それぞれの方式での実装イメージは以下の通り。

- DML インクリメンタル
  - INSERT/DELETE/UPDATE の executor: 各 DML 実行時に統計値を更新する
    - R(T): カウンタの増減 (+1/-1)
    - B(T), H(I): B+Tree のページ分割/マージ時にカウンタを更新 → B+Tree 内部の変更が必要
    - V(T, F): ユニーク値の Set をメモリ上に保持し、DML ごとに追加/削除する
    - min/max: INSERT 時は現在値と比較するだけ。DELETE 時は削除した値が min/max だった場合に再計算が必要 → 結局テーブル走査が発生する

- 閾値ベース ANALYZE
  - INSERT/DELETE/UPDATE の executor: `dirty_count++` するだけ
  - Optimizer: コスト計算前に `dirty_count` が閾値を超えていれば `analyze()` を呼ぶ
  - analyze(): テーブルの B+Tree をリーフ先頭から末尾まで 1 回走査し、全統計値 (R(T), B(T), V(T, F), min/max) を一括計算して保存する

結果、DML インクリメンタルは `V(T, F)` と DELETE 時の min/max の管理が困難で、B+Tree 内部にも変更が必要なのに対し、閾値ベース ANALYZE は DML 側が `dirty_count++` だけで済み、統計計算が 1 箇所に集約される。また、閾値ベース ANALYZE は DML のコードに統計更新ロジックが混入しないため、関心の分離としてもシンプルになる。

そのため、閾値ベース ANALYZE の方が実装コストが低いと思われる

#### 将来的な拡張性

- DML インクリメンタル: 新しい統計値が必要になるたびに、INSERT/DELETE/UPDATE すべての executor に更新ロジックを追加する必要がある
- 閾値ベース ANALYZE: 走査ループ内に計算を追加するだけで対応可能

よって、閾値ベース ANALYZE の方が拡張しやすいと思われる

#### 機能の必要性の範囲

それぞれの方式での機能的な優位点は以下の通り

- DML インクリメンタル
  - 統計情報が常に最新の状態に保たれるため、クエリ最適化の精度が高くなる
- 閾値ベース ANALYZE
  - 統計情報の収集回数は DML インクリメンタルの方式よりも少なくなるため、パフォーマンスへの影響は小さくなる可能性がある

ただし、minesql はあくまで学習目的であり、リアルタイム性やパフォーマンスへの影響などはあまり重要ではない。そのため、機能の必要性の範囲に関してはどちらでも大きな差はないと思われる

### 結論

実装コストや将来的な拡張性の観点では、閾値ベース ANALYZE の方式の方が適しており、機能の必要性についてはどちらでも大きな差はないと思われる

よって、閾値ベース ANALYZE で統計情報を収集する方式を採用することに決定した
