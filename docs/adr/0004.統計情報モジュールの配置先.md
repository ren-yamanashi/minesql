# 統計情報モジュールの配置先

- status: Approved
- date: 2026-03-23

## Motivation

[ADR 0002 - OptimizerとPlannerの分離](./0002.OptimizerとPlannerの分離.md) にて、コストベースの実行計画作成を、閾値ベース Analyze の方針で実装することが決定した。  

その閾値ベース Analyze の実装にあたり、以下の 2 つの関心事をどのパッケージに配置するか決める必要があった

- 統計情報の収集 (Analyze): テーブルをスキャンして R(T), V(T, F), min/max 等を算出する処理
- ステート管理 (dirty_count, last_analyze_row_count): Analyze の実行要否を判定するためのテーブルごとの状態

## Decisions

統計情報の収集とステート管理の両方を独立した statistics パッケージ (`internal/statistics`) に配置する

## Context

_前提_

dirty_count と last_analyze_row_count はディスクに永続化せずメモリ管理とする。  
(minesql はシングルプロセス前提であり、再起動時に統計情報が消えても次のクエリ最適化時に Analyze が走るだけなので、永続化のコスト対効果が薄いため)

### 候補の検討

統計情報の収集 (Analyze) の配置先として以下を検討した。

- planner: Analyze の実体はテーブルスキャンであり、planner の責務 (AST → 実行計画の変換) とは性質が異なる
- executor: executor は実行計画に基づいてクエリを実行する役割であるが、Analyze は実行計画ではない
- catalog: 静的なスキーマ定義の管理が責務であり、テーブルスキャンを伴う動的な処理は責務から外れる

また、ステート管理 (dirty_count, last_analyze_row_count) の配置先についても検討した。

- executor: executor は実行計画をステートレスに処理する役割であり、テーブルごとの長寿命なステートを持つのは責務的に合わなさそう
- engine: ストレージ層のリソースを管理しているが、統計情報の判定状態を持つのは engine の責務 (ストレージ層のリソース管理) から外れる

統計収集とステート管理は密接に関連しており (閾値判定には dirty_count と last_analyze_row_count の両方が必要)、分散させるとモジュール間の結合が増える。また、責務を考えた場合でも上述したどの配置先も微妙であった。

### 決定

独立した statistics パッケージ (`internal/statistics`) を新たに作成し、統計収集とステート管理の両方を配置する。

statistics パッケージはシングルトンとしてインスタンスを保持し、dirty_count や last_analyze_row_count をメモリ上で管理する。
