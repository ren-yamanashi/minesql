# Optimizer と Planner を分離するか

- status: Approved
- date: 2026-03-23

## Motivation

Planner にコストベースの最適化機能を追加するにあたり、Planner と Optimizer の責務の違い、struct やモジュールの分割方針を決める必要があった

## Decisions

現状では Planner と Optimizer を分離せず、Planner 内でコストベースの最適化機能を実装する。

将来的に JOIN をサポートして論理最適化が必要になったら、その時点で分離を検討する。

## Context

- Planner と Optimizer を分離する意味が出てくるのは、論理最適化 と 物理最適化 を分けたい場合
  - 論理最適化: AST (や論理プラン) を等価な形に書き換える (例: サブクエリの展開、不要な JOIN の除去など)
  - 物理最適化: アクセス方法やコストを比較して物理的な実行計画を選ぶ (例: テーブルスキャン vs インデックススキャン)
- minesql で現状やりたいのは物理最適化 (スキャン方法の選択) だけなので、AST を書き換えるフェーズが不要 (現状は JOIN をサポートしてないというのもあり)

これらの理由から、現状は Planner と Optimizer を分離せず、Planner 内でコストベースの最適化機能を実装する方針とする。

具体的な処理の流れは以下のイメージ

1. Planner で AST を受け取る
2. コストを比較してスキャン方法を決定
3. Executor を組み立てて返す

※ただし、将来 JOIN をサポートして論理最適化が必要になったら、その時点で分離を検討する
