# REDO ログのページ変更記録方式

## Motivation

REDO ログの実装にあたり、ページに対する変更内容をどのような形式で記録するかを決める必要がある。

## Decisions

ページ全体のコピー (4096 バイト) を REDO ログレコードに記録する方式を採用する。リカバリ時はページをそのまま上書きする。

## Context

以下の 3 つの方式が候補として挙がった。

- ページ全体のコピー
  - 変更後のページ全体 (4096 バイト) をそのまま REDO ログレコードに記録する
  - リカバリ時はページを上書きするだけで済む
- Before/After diff
  - ページの変更前にスナップショットを取り、変更後との差分 (変更された offset とバイト列) を記録する
- Physiological logging (InnoDB 方式)
  - 操作ごとに専用の REDO レコード型 (`MLOG_COMP_REC_INSERT`, `MLOG_2BYTES` 等) を定義し、ページに対する論理的な操作を記録する

これらを以下の基準で評価した。

- 実装の容易さ: 既存の BTree 層への変更が少なく、シンプルに実装できるか
- ログサイズの効率: REDO ログの容量がどの程度になるか
- 既存アーキテクチャとの親和性: 現在の `GetWriteData()` が生の `[]byte` を返す設計と整合するか

| 方式 | 実装の容易さ | ログサイズ | 既存アーキテクチャとの親和性 |
| --- | --- | --- | --- |
| ページ全体のコピー | 高い (変更後のページをそのまま記録) | 大きい (1 回の変更につき 4096 バイト) | 高い (既存の設計変更が不要) |
| Before/After diff | 中程度 (スナップショット管理と diff 計算が必要) | 小さい (変更箇所のみ記録) | 中程度 (スナップショット取得のフックが必要) |
| Physiological logging | 低い (BTree の全操作にレコード型が必要。InnoDB は数十種類) | 最小 (操作の種別とパラメータのみ記録) | 低い (`GetWriteData()` の設計を大きく変更する必要あり) |

minesql では BTree 層が `GetWriteData()` で生の `[]byte` を返し、呼び出し側が直接バイトを操作する設計のため、個別の変更をキャプチャする仕組みがない。Physiological logging や diff 方式を実現するにはこの設計を大きく変更する必要がある。ページ全体のコピー方式はログサイズが大きくなるが、minesql の規模では問題にならず、実装のシンプルさを優先してこの方式を採用した。
