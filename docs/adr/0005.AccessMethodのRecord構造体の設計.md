# Access Method の Record 構造体の設計

- status: Approved
- date: 2026-03-26

## Motivation

トランザクション機能の実装に向けて、access method が B+Tree に格納するレコードに `DeleteMark`、`LastModified`、`RollPtr` といったデータを付加する必要が生じた

現状のレコード表現は `type Record [][]byte` (全カラム値のフラット配列) であり、上記のデータを表現する手段がない  
そこで access method が受け取る Record の型をどのように設計するかを検討した

## Decisions

案 3 (案 1 と案 2 の折衷) を採用する

## Context

### 前提

- B+Tree は汎用的なデータ構造であり、レコードの `DeleteMark` 等を関知しない
- `DeleteMark` 等のデータは access method が B+Tree のレコードの非キーフィールドにエンコードする
- SELECT の戻り値には `DeleteMark` や `LastModified`、`RollPtr` は含まれない

### 案の概要

以下の 3 案を検討した

- 案 1: Record を構造体にする
  - executor 側と access method 側でそれぞれ別の Record 型を用意し、Key / NonKeyColumns の分離や `DeleteMark` 等の構造を型で表現する
- 案 2: Record をフラットに保つ
  - executor 側は現状の `[][]byte` を維持し、access method 側のみフラットな構造体を導入する
- 案 3: 案 1 と案 2 の折衷
  - executor 側は現状の `[][]byte` を維持しつつ、access method 側は案 1 と同じ構造化された Record を使用する

### 案 1: Record を構造体にする

_executor 側_ (executor パッケージ内で定義)

```go
type Record struct {
    Key          [][]byte // キーカラム値
    NonKeyColumns [][]byte // 非キーカラム値
}
```

_access method 側_ (access パッケージ内で定義)

```go
type RecordHeader struct {
    DeleteMark uint8
}

type RecordNonKey struct {
    LastModified uint64
    RollPtr      uint64
    NonKeyColumns [][]byte
}

type Record struct {
    Header RecordHeader
    Key    [][]byte
    NonKey RecordNonKey
}
```

- planner が AST からカラム値を取り出し、Key / NonKeyColumns に分離して `executor.Record` を組み立て、executor に渡す
- executor が `executor.Record` を `access.Record` に変換し、Header や NonKey の `LastModified` / `RollPtr` をセットして access method に渡す
- access method が `access.Record` を受け取り、B+Tree のレコードにエンコードする

### 案 2: Record をフラットに保つ

_executor 側_ (executor パッケージ内で定義)

```go
type Record [][]byte // 全カラム値のフラット配列 (現状と同じ)
```

_access method 側_ (access パッケージ内で定義)

```go
type Record struct {
    DeleteMark   uint8
    LastModified uint64
    RollPtr      uint64
    Columns      [][]byte // 全カラム値 (key + non-key) をフラットに格納
}
```

- planner は現状通り全カラム値をフラットに格納した `[][]byte` を組み立てる
- executor が `[][]byte` を `access.Record` に変換し、`DeleteMark` 等をセットして access method に渡す
- access method が `access.Record` を受け取り、`Columns` を `PrimaryKeyCount` で Key / NonKeyColumns に分割してから B+Tree のレコードにエンコードする

### 案 3: 案 1 と案 2 の折衷

案 1 の構造化された `access.Record` (Header / Key / NonKey) を採用しつつ、planner は案 2 と同様に `[][]byte` のまま維持する。Key / NonKeyColumns の分離は executor が行う

_executor 側_ (executor パッケージ内で定義)

```go
type Record [][]byte // 全カラム値のフラット配列 (現状と同じ)
```

_access method 側_ (access パッケージ内で定義)

```go
type RecordHeader struct {
    DeleteMark uint8
}

type RecordNonKey struct {
    LastModified uint64
    RollPtr      uint64
    NonKeyColumns [][]byte
}

type Record struct {
    Header RecordHeader
    Key    [][]byte
    NonKey RecordNonKey
}
```

- planner は現状通り全カラム値をフラットに格納した `[][]byte` を組み立てる
- executor が `[][]byte` を `access.Record` に変換する
  - `access.TableAccessMethod` の `PrimaryKeyCount` を使って Key / NonKeyColumns を分離し、`DeleteMark` 等もセットする
  - executor は既に `access.TableAccessMethod` を保持しているため、新たな依存は発生しない
- access method が `access.Record` を受け取り、B+Tree のレコードにエンコードする

---

### 検討項目

3 案を以下の検討項目に基づいて比較した

- レイヤー間の責務の明確さ
- 型安全性
- 将来の拡張性
- コードの可読性

### 検討項目ごとの比較

#### レイヤー間の責務の明確さ

各レイヤーの責務は以下の通り

- planner: AST → 実行計画の変換
- executor: 実行計画に基づいた処理の実行
- access method: ディスク上のデータ構造の読み書き (B+Tree のレコードへのエンコード / デコードも含む)

B+Tree のレコードへのエンコードは access method の責務。`DeleteMark` や `LastModified` のセットは実行時の処理であり executor の責務。Key / NonKeyColumns の分離は物理構造に関する処理であり、少なくとも planner が行うべきではない (planner の責務は AST → 実行計画の変換であり、物理構造の知識を持つべきではない)

- 案 1: planner が Key / NonKeyColumns の分離を行うため、planner が物理構造を意識する必要がある
- 案 2: Key / NonKeyColumns の分離も B+Tree のレコードへのエンコードも access method が行う。planner は物理構造を意識しない
- 案 3: executor が Key / NonKeyColumns の分離を行い、access method が B+Tree のレコードへのエンコードを行う。planner は物理構造を意識しない
  - executor は既に `access.TableAccessMethod` を保持しており `PrimaryKeyCount` にアクセスできるため、分離の処理を自然に行える

_結論_: 案 2 または 案 3 が適切。案 1 は planner に物理構造の責務が漏れるため不適切

#### 型安全性

`access.Record` において Key と NonKeyColumns が型レベルで分離されているかどうか。

- 案 1: Key と NonKeyColumns が別フィールドに分離されている
  - Header / NonKey の分離により `DeleteMark` 等と NonKeyColumns の混同も起きにくい
- 案 2: `Columns` はフラットな `[][]byte` であるため、Key / NonKeyColumns の境界は `PrimaryKeyCount` 依存
- 案 3: 案 1 と同等

_結論_: 案 1 または 案 3 が優れている

#### 将来の拡張性

トランザクション機能の本格実装時に、Record に新しいフィールドを追加する際の見通しの良さ

- 案 1: 構造体が物理レイアウトを反映しているため、Header や NonKey にフィールドを追加しやすい
- 案 2: `access.Record` にフィールドを足すだけで済むが、フラットなためフィールドが増えると見通しが悪くなりやすい
- 案 3: 案 1 と同等

_結論_: 案 1 または 案 3 が優れている

#### コードの可読性

`access.Record` を扱うコードから、各フィールドの意味が読み取れるかどうか

- 案 1: Record を扱うコードから、Key / NonKeyColumns / `DeleteMark` 等の意味が明確に読み取れる
- 案 2: Record を扱うコードはシンプルだが、`Columns` の中身の構造は暗黙的
- 案 3: 案 1 と同等

_結論_: 案 1 または 案 3 が優れている

---

### 検討結果のまとめ

| 検討項目 | 案 1 (構造体) | 案 2 (フラット) | 案 3 (折衷) |
| --- | --- | --- | --- |
| レイヤー間の責務の明確さ | planner に物理構造の責務が漏れる | 適切 | 適切 |
| 型安全性 | 高い | 現状と同等 | 高い |
| 将来の拡張性 | 見通しが立てやすい | 見通しが劣る可能性あり | 見通しが立てやすい |
| コードの可読性 | 意味が明確 | 暗黙的 | 意味が明確 |

### 結論

案 1 は型安全性・可読性・拡張性の面で優れているが、planner が Key / NonKeyColumns の分離を行うため物理構造を意識する必要がある。planner の責務は AST → 実行計画の変換であり、物理構造に踏み込むのは適切でない

案 2 は planner が物理構造を意識せずに済むが、`access.Record` がフラットなため型安全性・可読性・拡張性の面で案 1 に劣る

案 3 は案 1 の構造化された `access.Record` (Header / Key / NonKey) を採用しつつ、planner は `[][]byte` のまま維持する。Key / NonKeyColumns の分離は executor が `access.TableAccessMethod.PrimaryKeyCount` を使って行う。executor は既にこのフィールドにアクセスできるため新たな依存は発生しない。これにより、案 1 の利点 (型安全性・可読性・拡張性) と案 2 の利点 (planner が物理構造を意識しない) を両立できる

以上から、案 3 を採用する

## Result

[ADR 0006](./0006.セカンダリインデックスのソート順の実現方法.md) で B+Tree のデータ構造を key-value ペアからレコードに変更することが決まり、レコードの物理レイアウトにヘッダー領域を設けることになった

`access.Record` の構造に変更はない。`RecordHeader` (`DeleteMark`) は B+Tree のレコードヘッダーに、`RecordNonKey` (`LastModified`, `RollPtr`, `NonKeyColumns`) は非キー領域にそれぞれ対応する
