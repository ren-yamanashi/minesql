# アクセスメソッド

## 参考文献

- [The physical structure of records in InnoDB](https://blog.jcole.us/2013/01/10/the-physical-structure-of-records-in-innodb/)
- [InnoDB Row Formats - MySQL 8.0 Reference Manual](https://dev.mysql.com/doc/refman/8.0/en/innodb-row-format.html)
- [InnoDB ソースコード: dict_index_build_internal_clust (dict0dict.cc)](https://github.com/mysql/mysql-server/blob/8.0/storage/innobase/dict/dict0dict.cc) - クラスタ化インデックスのフィールド順序定義

## 概要

- データ構造を辿って要求に応じたデータ操作を行う
  - 実際のデータの読み書きはバッファプールに依頼する
- アクセスメソッドのデータ構造とアルゴリズムには [B+Tree](../btree/btree.md) を使用

## テーブル

- テーブルに対して1 つの `${table_name}.db` ファイルが存在する
- 1 つのテーブルには 1 つのクラスタ化インデックスと複数のセカンダリインデックスが存在する (セカンダリインデックスは存在しない場合もある)

### クラスタ化インデックス

- プライマリキーはクラスタ化インデックスとする
- テーブルの実態は、プライマリキーをキー領域、その他カラム値を非キーフィールドとするレコード (後述)
- プライマリキーが複合キーの場合も考慮して、複合キーの場合も正しくソートされるように [Memcomparable format](../encode/memcomparable-format.md) を使用してプライマリキーをエンコードする

### セカンダリインデックス

- 非 UNIQUE インデックスにも対応可能にするため、セカンダリインデックスのキーは `セカンダリキー + プライマリキー` の構成にしている
  - (B+Tree のキーに対して一意制約を設けた方が実装が楽なため)
- B+Tree のソート順は (セカンダリキー, プライマリキー)
- 検索の際は、まずセカンダリインデックスを検索し、その後必要があればキー領域に含まれるプライマリキーを使用して、クラスタ化インデックスを辿ることでレコード情報を取得する

## レコード

- B+Tree の[レコード](../btree/node/node.md#レコード)はヘッダー領域・キー領域、非キー領域の 3 領域で構成される
- アクセスメソッドは、それらの領域をインデックスの種類に応じて以下のように意味づける
- B+Tree のソートはキーに基づいて行われる
- レコードはそれぞれ「どこからどこまでがキーか」という情報を持っているため、それに基づいてソート順が決定される

#### クラスタ化インデックス

| 領域 | フィールド | バイト数 | 説明 |
| --- | --- | --- | --- |
| ヘッダー | DeleteMark | 1 | 削除マーク |
| キー | プライマリキー | 可変 | プライマリキーのカラム値をエンコードしたもの |
| 非キー | lastModified | 8 | このレコードを最後に INSERT/UPDATE したトランザクション ID |
| 非キー | rollPtr | 4 | undo ログレコードへのポインタ |
| 非キー | 非キーカラム | 可変 | プライマリキー以外のカラム値をエンコードしたもの |

#### セカンダリインデックス

| 領域 | フィールド | バイト数 | 説明 |
| --- | --- | --- | --- |
| ヘッダー | DeleteMark | 1 | 削除マーク |
| キー | セカンダリキー | 可変 | セカンダリインデックスを構成するカラム値をエンコードしたもの |
| キー | プライマリキー | 可変 | プライマリキーを構成するカラム値をエンコードしたもの |

## 操作

### レコードの挿入

- クラスタ化インデックスの B+Tree にレコードを挿入
  - 同一プライマリキーのレコードがすでに論理削除済み (DeleteMark=1) の場合はそのレコードを上書きし、DeleteMark を 0 に戻す
- セカンダリインデックスにも同様に挿入する (UNIQUE インデックスであれば、重複確認を行う)

### レコードの削除

- レコードを論理削除 (DeleteMark=1) する
- 物理削除は後でパージによって行われる
- クラスタ化インデックス: レコード内の DeleteMark を 1 に更新
- セカンダリインデックス: レコード内の DeleteMark を 1 に更新
- レコードの検索時には、DeleteMark = 1 のレコードをスキップする

### レコードの更新

- クラスタ化インデックスにおけるレコードの更新は、キー (=プライマリキー) が変わるかどうかで処理が異なる
  - キーが変わらない場合: インプレース更新
  - キーが変わる場合: 論理削除 + 挿入
- セカンダリインデックスにおけるレコードの更新は、常に論理削除 + 更新で実現する
  - これは MySQL の InnoDB の方針に従っている (参照: [InnoDB のマルチバージョニング](https://dev.mysql.com/doc/refman/8.0/ja/innodb-multi-versioning.html) の「マルチバージョニングおよびセカンダリインデックス」の項目)

#### セカンダリインデックスの更新について

- セカンダリインデックスのキー領域は `セカンダリキー + プライマリキー` で構成されるため、論理削除後に同じセカンダリキーで新しいレコードを挿入しても、B+Tree 上のキー重複は起こらない
- テーブル `users (id PK, name, email UNIQUE)` で `email = 'alice@example.com'` の行の `id` を `1` → `2` に更新する場合の更新の流れは以下のようになる
  1. 論理削除: `key=(alice@example.com, 1)` の DeleteMark を 1 にセット
  2. 挿入: `key=(alice@example.com, 2)` を DeleteMark 0 で挿入
- ユニーク制約のチェックはセカンダリキー部分 (`alice@example.com`) のみに対して行い、また論理削除済みのレコードはスキップする
