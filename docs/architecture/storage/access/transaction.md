# トランザクション

## 参考文献

[MySQL 8.0 リファレンスマニュアル - InnoDB マルチバージョン](https://dev.mysql.com/doc/refman/8.0/ja/innodb-multi-versioning.html)

## 概要

- MineSQL では、MVCC によってトランザクション管理を実装している
- MVCC はデータの複数バージョンを保持することで読み取りと書き込みの競合を回避する同時時効制御方式
  - Strict 2PL はデータの一貫性を保証するが、読み取りと書き込みが違いにブロックし合うので、並行性が制限される (読み取り同士はブロックしないが、読み取りと書き込みはブロックする)
  - MVCC は読み取りと書き込みが同時に行えるようにすることで、並行性を向上させる
    - 書き込み同士はブロックするが、読み取りと書き込みはブロックしない

### スナップショット (Read View)

- MVCC では、トランザクションが「どのデータが見えるか」を決定するためにスナップショット (`ReadView`) を使用する
- スナップショットを取得するタイミングは、トランザクション分離レベルによって異なる
  - READ COMMITTED: 各ステートメント実行時にスナップショットを取得する
  - REPEATABLE READ: トランザクション開始時にスナップショットを取得し、終了まで保持する

スナップショットは以下の情報を持つ

| 項目 | 説明 |
| --- | --- |
| `trxId` | 自分のトランザクション ID |
| `mLowLimitId` | コミット済みの最大の `trxId` |
| `mUpLimitId` | このトランザクションの次に払い出される `trxId` |
| `mIds` | まだコミットされていない `trxId` のリスト |

### 行の構造

- データベース内に格納された各レコードに対して以下のフィールドが追加される
  - `lastModified` 自分を最後に変更したトランザクション ID
  - `rollPtr` 自分が更新される 1 世代前のレコードの内容 (undo ログ) へのポインタ (=ロールバックポインタ)
  - `deleteMark` 削除済みかどうかを示すフラグ。DELETE 時に true に設定され、後でパージによって物理削除される

## 処理の流れ

### INSERT

1. そのレコードがどのページに配置されるべきかを計算
2. undo ログにレコードの挿入を記録する (ロールバック時にこのレコードを削除するため)
3. 書き込み対象のページがすでにバッファプールに読み取り済みであればそれを、バッファプールにない場合はディスクからページを吸い上げてそのページにレコードを書き込む。この際に `lastModified` に自分の `trxId` を書き込む (`rollPtr` にはポインタではなくフラグが立つ。自分の 1 世代前が存在しないため)

※ undo ログを先に書かないと、レコード挿入後にクラッシュした場合にロールバックできなくなってしまうため、レコードの書き込みの前に undo ログに書き込む必要がある

### SELECT

※単純化するために、プライマリキーをイコール検索したと仮定する

1. 目的のレコードを見つける
   - クラスタ化インデックスの B+Tree を辿って目的のノード (および対象のレコード) を見つける

2. 目的のレコードの `lastModified` を確認する
   - `lastModified` がない場合はそのレコードはコミットされていないので読んではいけない
   - トランザクション分離レベルが `READ COMMITTED` の場合
     - `lastModified` が記録されていれば、そのレコードはコミットされていると判断して読んでいいとする (READ COMMITTED の場合はファントムリードが発生しても問題ないため)
   - トランザクション分離レベルが `REPEATABLE READ` の場合
     - `lastModified` に記録された `trxId` が自分のトランザクションの開始時にコミット済みでない場合は読んではいけない (= `ReadView` の `mUpLimitId` より大きいか、または `mIds` に含まれている場合)
   - 読んではいけないレコードと判断された場合、読む側のトランザクションがそれを迂回して (`rollPtr` を使用して) 自分の読むべきレコードを探しにいく
     - 現在バッファプールに載っているレコードの `rollPtr` から、undo ログから「自分の 1 世代前のレコードの内容」を読み出す
     - ここで再度 `lastModified` の判定をし、最終的に自分が読めるレコードを見つけるか、最後まで見つけられなければ終了する

### UPDATE

1. SELECT と同様に、更新対象のレコードを見つける
   - ただし、更新対象のレコードを読んではいけないと判断した場合は、`rollPtr` を使用して探しに行かない
2. 更新対象のレコードを見つけたら、undo ログにレコードの更新を記録する
   - PK が変わらない場合は、undo ログに更新前のレコードの内容を記録する (インプレース更新)
   - PK が変わる場合は、該当レコードを削除するための undo ログと、更新後のレコードを挿入するための undo ログの両方を記録する (Delete + Insert)
3. 更新対象のレコードを更新し、`lastModified` に自分の `trxId` を、`rollPtr` に undo ログエントリへのポインタを書き込む

### DELETE

1. SELECT と同様に、削除対象のレコードを見つける
   - ただし、削除対象のレコードを読んではいけないと判断した場合は、`rollPtr` を使用して探しに行かない
2. 削除対象のレコードを見つけたら、undo ログに削除前のレコードの `rowId` とレコードの内容を記録する
3. 削除対象のレコードを削除する
   - ただし、実際にレコードを物理削除するのではなく、削除対象のレコードに `deleteMark` フラグを立てる
