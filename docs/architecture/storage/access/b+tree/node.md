# Node

## ノードの種類

- [リーフノード](./leaf_node.md)
- [ブランチノード](./branch_node.md)

## ノード構造

- すべてのノードは、先頭に 8 バイトのノードタイプヘッダーを持つ
  - ノードタイプヘッダー: 8 バイト
    - `"LEAF    "` (リーフノード) または `"BRANCH  "` (ブランチノード)
    - ページデータの先頭 8 バイトを読み取ることで、ノード型を判定できる

## key-value ペア

- ノードは key-value ペアを格納する。
- ペアのバイト列は以下のように構成される

  ```txt
  |key サイズ  |key (可変長)      |value (可変長)      |
  ↑__4 bytes__↑__keySize bytes__↑__valueSize bytes__↑
  ```

## 指定したキーを持つペアを見つける方法

### 前提

- ノード内には複数の key-value ペアが格納されている
- ノード内のペアはキーの昇順でソートされている
- ノードはペア数を保持している (自分のノードの中にいくつのペアが存在するかの情報)
- 1 ノード = 1 ページ = 4096 バイト である
  - 特定の一つのページに複数の B+Tree ノードのデータが格納されることはない
  - MySQL の InnoDB においてもおそらくその設計 (1ページ=1ノード) になっている
    - 参考: https://planetscale.com/blog/btrees-and-database-indexes

### 探索の流れ

1. ノード内のペア数を取得する
2. `0 ~ ペア数` の範囲で、ノード内のペアを二分探索する
   - ペア (二分探索でいうところの mid に位置するペア) のキーと、検索したいキーを比較する
   - 比較の結果
     - 一致: 目的のペアが見つかったと判断する
     - ペアのキー < 検索したいキー: 探索範囲を右半分に絞る
     - ペアのキー > 検索したいキー: 探索範囲を左半分に絞る
3. ペアが見つかるか、探索範囲がなくなるまで `2` の流れを繰り返す
   - 最後まで目的のペアが見つからなければ、ノード内にそのキーは存在しないと判断する
     - 最後まで: left >= right になるまで (left と right が徐々に近づいていき、最終的に同じ位置を指すまたは交差するまで)
