# ページクリーナー

## 概要

- データ変更によって生じたダーティーページを定期的にディスクにフラッシュする仕組み
- ダーティーページのフラッシュにより[チェックポイント](./checkpoint.md)の checkpoint LSN が進み、不要な REDO ログを切り詰められるようになる
- また、同じページに対して複数回の更新があった場合に 1 回のフラッシュで済むため、ディスク I/O を削減できる

## フラッシュリスト

ページクリーナーがどのページをフラッシュすべきかを管理するために、バッファプール内にフラッシュリストを持つ (InnoDB に合わせている)

- ページが最初にダーティーになった時点でフラッシュリストに追加する
- 既にフラッシュリストに含まれるページが再度変更された場合は追加しない (リスト内の順序を維持する)
- フラッシュリストはダーティーになった順に並んでおり、古いものほどリストの先頭に位置する
- ページがフラッシュされたらフラッシュリストから除外する

この順序管理により、ページクリーナーは古いダーティーページから優先的にフラッシュでき、checkpoint LSN を効率的に進めることができる。

## フラッシュの流れ

1. フラッシュリストの先頭 (最も古いダーティーページ) から順にフラッシュ対象を選択する
2. 対象ページの Page LSN 以上の REDO ログがフラッシュ済みであることを確認する
3. ページをディスクに書き出し、フラッシュリストから除外する
4. フラッシュ後、checkpoint LSN を更新する

## フラッシュの契機

- REDO ログのサイズが閾値を超えた場合
- バッファプールのダーティーページ率が閾値を超えた場合

| 環境変数 | 説明 | デフォルト値 |
| --- | --- | --- |
| `MINESQL_REDO_LOG_MAX_SIZE` | REDO ログの最大サイズ (バイト) | 1048576 (1MB) |
| `MINESQL_MAX_DIRTY_PAGES_PCT` | ダーティーページ率の上限 (%) | 90 |
