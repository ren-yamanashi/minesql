# REDO ログ

## 参考文献

- [InnoDB のリカバリ](https://dev.mysql.com/doc/refman/8.0/ja/innodb-recovery.html)

## 概要

- REDO ログは、トランザクションの変更内容を記録するログ
- バッファプールに変更内容を書き込む前に、必ずその変更記録を REDO ログに書き込む (Write-Ahead Logging)
  - 詳細: [Durability を実現するためのアルゴリズム](../../../about/durability.md#write-ahead-logging-wal)

### REDO ログレコード

- REDO ログレコードには以下の情報が含まれる

| 項目 | 説明 |
| --- | --- |
| トランザクション ID | 変更を行ったトランザクションの ID |
| LSN (Log Sequence Number) | 詳細: [Log Sequence Number (LSN)](../../../about/durability.md#log-sequence-number-lsn) |
| 変更内容 | どのデータアイテムに対して、どのような変更を行ったか |

## Page LSN

- 各ページは [Slotted Page ヘッダー](../btree/node/slotted-page.md#ヘッダーの構成) の予約領域 (4 バイト) を Page LSN として使用する
- Page LSN は、そのページに最後に適用された REDO ログレコードの LSN を記録する
- クラッシュリカバリ時、REDO ログレコードの LSN がページの Page LSN 以下であれば、そのレコードは適用済みなのでスキップする (冪等性の保証)

## クラッシュリカバリ

- サーバーを正常終了した場合、minesql ではすべてのダーティーページをバッファプールからディスクにフラッシュするのを待機してから終了する
  - サーバー終了時にコミットされていなかったトランザクションはロールバックされる
- 従って、正常終了している限りは、それ以前のコミットはすべてディスクに同期されたクリーンな状態になる
- 逆に異常終了した場合は、ダーティーページの存在を留意する必要がある
- これをケアするための異常終了直後の起動時に走るディスクファイルに整合性を戻すための処理を「クラッシュリカバリ」と呼ぶ

### クラッシュリカバリの流れ

異常終了後の再起動時、以下の 2 ステップでデータの整合性を復元する

#### Step 1: REDO ログの適用

- REDO ログを先頭からスキャンし、記録された変更をページに再適用する
- これにより、コミット済みだがディスクに未反映だった変更が復元される
- [Page LSN](#page-lsn) を参照し、適用済みのレコードはスキップする (冪等性の保証)
- 正常終了時にすべてのダーティーページがフラッシュされていた場合、REDO ログの適用はスキップされる

#### Step 2: 不完全なトランザクションのロールバック

- REDO ログの適用後、クラッシュ時にコミットされていなかったトランザクションを特定する
- 該当するトランザクションの UNDO ログを逆順に適用し、未コミットの変更を取り消す
- これにより、REDO ログの適用で復元された「コミットに失敗したトランザクションの変更」が正しく取り消される

### ケースごとの挙動

#### トランザクション $T_1$ が COMMIT した後にクラッシュした場合

- REDO ログには $T_1$ の変更内容と COMMIT レコードが記録されている
- Step 1 で REDO ログを適用し、$T_1$ の変更がページに反映される
- Step 2 では $T_1$ は COMMIT 済みなのでロールバック対象にならない
- 結果: $T_1$ の変更は保持される

#### トランザクション $T_1$ が ROLLBACK した後にクラッシュした場合

- ROLLBACK 時に UNDO ログで変更が取り消され、REDO ログにはその取り消し操作も記録されている
- Step 1 で REDO ログを適用すると、ROLLBACK 後の状態が復元される
- Step 2 では $T_1$ は既に ROLLBACK 済みなのでロールバック対象にならない
- 結果: $T_1$ の変更は存在しない (ROLLBACK が反映された状態)

#### トランザクション $T_1$ が COMMIT/ROLLBACK する前にクラッシュした場合

- REDO ログには $T_1$ の変更内容が記録されているが、COMMIT レコードも ROLLBACK レコードも存在しない
- Step 1 で REDO ログを適用すると、$T_1$ の未コミットの変更もページに反映される
- Step 2 で $T_1$ は COMMIT レコードも ROLLBACK レコードもないため未完了と判定され、UNDO ログを使ってロールバックされる
- 結果: $T_1$ の変更は取り消される
