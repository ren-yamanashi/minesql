# ページ

## 概要

- ディスク I/O の最小単位
- サイズは 16,384 バイト (16KB)
  - 理由
    - ファイルはファイルシステムによって HDD や SSD などのデバイスに書き込まれるが、この際にブロックサイズ以下の読み書きは、ファイルシステムによって勝手にブロックサイズ単位に切り上げられてしまう
    - そのためアプリケーション (RDB) が書き込み単位をブロックサイズに切り上げてもほとんど同じになる
    - Linux の一般的なファイルシステムである `ext4` のデフォルトのブロックサイズが 4096 バイト (4KB) であるため、その整数倍にしたいというのと、MySQL のデフォルトのページサイズが 16KB なので、MineSQL でも 16KB にしている
- 全ページ型に共通するヘッダーとボディ (ページ型固有のデータ) を持つ

## レイアウト

| オフセット | サイズ | フィールド | 説明 |
|----------|--------|-----------|------|
| 0 - 3    | 4 バイト | ヘッダー | 全ページ型共通のヘッダー領域 |
| 4+       | 16,380 バイト | ボディ | ページ型固有のデータ領域 |

## PageId

- ページがどのファイルのどの位置にあるかを特定するための識別子

| フィールド | バイト数 | 説明 |
| --- | --- | --- |
| FileId | 4 | ディスクファイルの識別子 <br/> 全体で一意 |
| PageNumber | 4 | ファイル内のページ番号 <br/> ファイル内で一意 |

_補足_

- PageId はバッファプールでも使用する識別子であるため、全体で一意な値である必要がある
- 例えば `PageId = ファイル内のページ識別子` (つまり全体で一意な値ではなく、あくまでファイル内で一意な値) とすると、バッファプールでは PageId をそのまま使用できない
  - なぜなら複数のファイルのページを扱うため、PageId だけではどのファイルのページなのか特定できないから
- そのため、PageId は "ファイル番号 + ファイル内のページ番号" で構成し、全体で一意な値としている
