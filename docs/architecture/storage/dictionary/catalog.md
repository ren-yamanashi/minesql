# カタログ

## 概要

- テーブルのメタデータ (テーブル名、カラム情報、インデックス情報など) を管理する
- カタログは、実データファイル (`${table_name}.db`) とは別の `minesql.db` というファイルに保存される
  - MySQL v8.0 以降でいうところのデータディクショナリテーブル (`mysql.idb`)
- カタログは以下の 6 つの B+Tree で構成される
  - テーブルメタデータ
  - インデックスメタデータ
  - インデックスキーカラムメタデータ
  - カラムメタデータ
  - 制約メタデータ
  - ユーザーメタデータ

### ヘッダーページ

- カタログファイルの先頭ページはヘッダーページとしている
- ヘッダーページの構成は以下

| オフセット | バイト数 | 説明 |
| --- | --- | --- |
| 0 | 3 | ファイルシグネチャとして設定されるマジックナンバー (固定値 `MINE` が格納される) |
| 4 | 3 | テーブルメタデータの B+Tree のメタページの PageNumber |
| 8 | 3 | インデックスメタデータの B+Tree のメタページの PageNumber |
| 12 | 3 | インデックスキーカラムメタデータの B+Tree のメタページの PageNumber |
| 16 | 3 | カラムメタデータの B+Tree のメタページの PageNumber |
| 20 | 3 | 制約メタデータの B+Tree のメタページの PageNumber |
| 24 | 3 | ユーザーメタデータの B+Tree のメタページの PageNumber |
| 28 | 3 | 次に割り当てる FileId |
| 32 | 3 | 次に割り当てるインデックス ID (グローバルで一意) |
| 36 | 3 | Undo ログファイル用の FileId |

## テーブルメタデータ

- B+Tree の[レコード](../btree/node/node.md#レコード)はヘッダー領域・キー領域・非キー領域の 3 領域で構成される
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-tables-table.html
- テーブルメタデータでは、各領域を以下のような内容で使用する (意味付けている)

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | テーブルメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | テーブル名 | - |
| 非キー | テーブルの属性情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - プライマリインデックスの B+Tree メタページ ID
  - テーブルのカラム数

## インデックスメタデータ

- インデックスメタデータでは、各領域を以下のような内容で使用する
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-indexes-table.html

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | インデックスメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + インデックス名 | インデックスが属するテーブルの FileId とインデックス名を組み合わせたもの |
| 非キー | インデックスの属性情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - インデックス ID (グローバルで一意)
  - インデックスの種類
    - 1: プライマリインデックス
    - 2: 一意なセカンダリインデックス
    - 3: 一意でないセカンダリインデックス
  - インデックスを構成するカラム数
  - インデックスの B+Tree メタページ ID

## インデックスキーカラムメタデータ

- インデックスを構成するカラム情報を管理する
- 1 つのインデックスは複数のカラムで構成される場合があるため、インデックスメタデータとは分けている
- インデックスメタデータに直接カラム情報を持たせると、インデックスの種類などの属性が各カラム行に冗長に繰り返される (部分関数従属になる)
  - 補足: インデックスメタデータのキーを `インデックス ID + カラム名` にすれば正規化違反は回避できるが、インデックス自体の属性 (種類など) を取得するためだけにカラム数分の行を走査する必要があるのでそれは微妙
- インデックスキーカラムメタデータでは、各領域を以下のような内容で使用する
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-fields-table.html

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | インデックスキーカラムメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | インデックス ID + カラム名 | インデックスメタデータのインデックス ID と構成するカラム名 (複合インデックスの場合はいずれか 1 つ) を組み合わせたものをキーとすることで、構成するカラム情報を一意に識別できるようにしている |
| 非キー | インデックスを構成するキーの情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - インデックス上のカラム位置

## カラムメタデータ

- カラムメタデータでは、各領域を以下のような内容で使用する
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-columns-table.html

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | カラムメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + カラム名 | カラムが属するテーブルの FileId とカラム名を組み合わせたものをキーとすることで、テーブル内でカラムを一意に識別できるようにしている |
| 非キー | カラムの順序位置などの情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - カラムの順序位置
- カラムメタデータには、すべてのテーブルのカラム情報が含まれる

※データはすべて文字列型

## 制約メタデータ

- 制約メタデータは PRIMARY KEY、UNIQUE KEY、FOREIGN KEY などの制約情報をカラム単位で管理する
- キーに `カラム名 + 制約名` を含むため、複合キー制約の場合は同じ制約名で複数行が存在する
  - 補足: インデックスメタデータとインデックスキーカラムメタデータは分けているのに、なぜ制約メタデータは分けていないのかについて
    - 制約の検索は「あるカラムにどんな制約があるか」の確認を目的とするため、対象テーブルのカラムを起点に行われる (=`FileId + カラム名` で検索)
    - よって制約名だけから属性を引くユースケースがなく、分離しても検索効率の改善にならないため、インデックスキーカラムメタデータのような分離はしてない
- 制約メタデータは、各領域を以下のような内容で使用する
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-key-column-usage-table.html

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | 制約メタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + カラム名 + 制約名 | 制約が属するテーブルの FileId、カラム名、制約名を組み合わせたものをキーとする |
| 非キー | 参照先の情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - 制約によって参照されるテーブルの FileId
    - 主キーやユニークキーの場合は無効値になる
  - 制約によって参照されるカラム名
    - 主キーやユニークキーの場合は空文字になる (NULL 未サポートのため)
- 主キーの場合、制約名は `PRIMARY` になる
- 外部キーだけでなく、カラムが主キーの場合、ユニークキーの場合なども制約として管理している

## ユーザーメタデータ

- ユーザーアカウントの認証情報を管理する

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | 使用しない |
| キー | ユーザー名 | ユーザーを一意に識別する |
| 非キー | ホスト名、認証文字列 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - ホスト名 (接続を許可するホスト)
  - 認証文字列 (`SHA256(SHA256(password))` の 32 バイト)
