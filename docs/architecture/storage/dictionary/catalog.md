# カタログ

## 概要

- テーブルのメタデータ (テーブル名、カラム情報、インデックス情報など) を管理する
- カタログは実データとは別のファイル (`minesql.db`) に保存される
  - MySQL v8.0 でいうところのデータディクショナリテーブル (`mysql.idb`)
- 実データは 各テーブルごとに `${table_name}.db` に保存される
- カタログは以下の 5 つの B+Tree で構成される
  - テーブルメタデータ B+Tree
  - インデックスメタデータ B+Tree
  - カラムメタデータ B+Tree
  - 制約メタデータ B+Tree
  - ユーザーメタデータ B+Tree

## ヘッダーページ

- カタログファイルの先頭ページはヘッダーページとしている
- ヘッダーページの構成は以下
  - オフセット 0-3: ファイルシグネチャとして設定されるマジックナンバー (固定値 `MINE` が格納される)
  - オフセット 4-7: テーブルメタデータの B+Tree のメタページ ID
  - オフセット 8-11: インデックスメタデータの B+Tree のメタページ ID
  - オフセット 12-15: カラムメタデータの B+Tree のメタページ ID
  - オフセット 16-19: 制約メタデータの B+Tree のメタページ ID
  - オフセット 20-23: ユーザーメタデータの B+Tree のメタページ ID
  - オフセット 24-27: 次に割り当てる FileId
  - オフセット 28-31: UNDO ログ用の FileId

## テーブルメタデータ

- [btree/node/record.md](../btree/node/record.md) に記載のとおり、B+Tree のレコードはヘッダー・キーフィールド・非キーフィールドの 3 領域で構成される
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-tables-table.html
- テーブルメタデータでは、各領域を以下のような内容で使用する (意味付けている)

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | テーブルメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId | テーブルの実データが格納されるディスクファイルの識別子 |
| 非キー | テーブル名などの情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - テーブル名
  - テーブルのカラム数

## インデックスメタデータ

- 同様に、インデックスメタデータでは、各領域を以下のような内容で使用する (意味付けている)
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-indexes-table.html

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | インデックスメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + インデックス名 | インデックスが属するテーブルの FileId とインデックス名を組み合わせたものをキーとすることで、テーブル内でインデックスを一意に識別できるようにしている |
| 非キー | インデックス種類などの情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - インデックスを構成するカラム数
  - インデックスの種類
    - 一意でないセカンダリインデックス
    - 一意なセカンダリインデックス
    - クラスタ化インデックス
- 複合インデックスはサポートしていない

## カラムメタデータ

- 同様に、カラムメタデータでは、各領域を以下のような内容で使用する (意味付けている)
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-innodb-columns-table.html

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | カラムメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + カラム名 | カラムが属するテーブルの FileId とカラム名を組み合わせたものをキーとすることで、テーブル内でカラムを一意に識別できるようにしている |
| 非キー | カラムの順序位置などの情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - カラムの順序位置
  - データ型
- 現状はカラムの型情報を保存しておらず、全て文字列型として扱っている

## 制約メタデータ

- 制約メタデータは、カラムの制約情報を表す
  - 参考: https://dev.mysql.com/doc/refman/8.0/ja/information-schema-key-column-usage-table.html
- 他のメタデータテーブルと同様に、各領域を以下のような内容で使用する (意味付けている)

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | 制約メタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + カラム名 + 制約名 | 制約が属するテーブルの FileId、カラム名、制約名を組み合わせたものをキーとする |
| 非キー | 参照先の情報 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - 制約によって参照されるテーブル名
    - 主キーやユニークキーの場合は空文字になる (NULL 未サポートのため)
  - 制約によって参照されるカラム名
    - 主キーやユニークキーの場合は空文字になる (NULL 未サポートのため)
- 主キーの場合、制約名は `PRIMARY` になる
- 外部キーだけでなく、カラムが主キーの場合、ユニークキーの場合なども制約として管理している
- 今の所「制約名から制約の詳細を引く」ユースケースがないが、将来的にそのユースケースが増えた場合には、インデックスの追加も検討する (インデックスがないと 全レコードの走査が必要になるため)

## ユーザーメタデータ

- ユーザーアカウントの認証情報を管理する

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | 使用しない |
| キー | ユーザー名 | ユーザーを一意に識別する |
| 非キー | ホスト名、認証文字列 (詳細は以下) | - |

- 非キー領域の内容は以下のとおり
  - ホスト名 (接続を許可するホスト。`%` で全ホスト許可、`192.168.1.%` でサブネットパターン)
  - 認証文字列 (`SHA256(SHA256(password))` の 32 バイト)

## 外部キー

- ON DELETE / ON UPDATE のアクションは RESTRICT のみサポートする
  - 参照先のレコードが参照元から参照されている場合、削除や更新を拒否する
- 自己参照外部キー (テーブルが自分自身を参照する外部キー) はサポートしない
- 外部キーカラム (子テーブル側) にはインデックス (KEY, UNIQUE KEY, PRIMARY KEY のいずれか) が必須
  - 自動作成はされず、明示的にインデックスを作成する必要がある (作成しないとエラーになる)
- 外部キーの参照先カラム (制約によって参照されるカラム) には PRIMARY KEY または UNIQUE KEY が必須
  - 参照先のカラムが一意であることを保証しないと、参照元レコードの更新時に不都合が出る

  - 参照先のカラムが一意でないと以下のような問題が発生する
    - 以下のようなテーブルがあると仮定

      ```sql
      -- もし非ユニークな参照先を許した場合
      users: | id | name  |
             | 1  | Alice |
             | 2  | Alice |
      orders: | id | user_name |  -- REFERENCES users(name)
              | 1  | Alice     |
      ```

      - `users.name` が非ユニークなため、`orders.user_name` の値 `Alice` がどちらの `users` レコードを参照しているのか特定できない
      - そのため、仮に `users` テーブルの `id=1` のカラムを削除した際に、`orders` テーブルの `id=1` のレコードがあるから RESTRICT で削除を拒否したら良いのか、それとも `users` テーブルの `id=2` のカラムがあるから削除可能として良いのか判断できない
