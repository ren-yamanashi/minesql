# カタログ

## 概要

- テーブルのメタデータ (テーブル名、カラム情報、インデックス情報など) を管理する
- カタログは実データとは別のファイル (`minesql.db`) に保存される
  - MySQL v8.0 でいうところのデータディクショナリテーブル (`mysql.idb`)
- 実データは 各テーブルごとに `${table_name}.db` に保存される
- カタログは以下の 3 つの B+Tree で構成される
  - テーブルメタデータ B+Tree
  - インデックスメタデータ B+Tree
  - カラムメタデータ B+Tree

## ヘッダーページ

- カタログファイルの先頭ページはヘッダーページとしている
- ヘッダーページの構成は以下
  - オフセット 0-3: ファイルシグネチャとして設定されるマジックナンバー (固定値 `MINE` が格納される)
  - オフセット 4-7: テーブルメタデータの B+Tree のメタページ ID
  - オフセット 8-11: インデックスメタデータの B+Tree のメタページ ID
  - オフセット 12-15: カラムメタデータの B+Tree のメタページ ID
  - オフセット 16-19: 次に割り当てる FileId

## テーブルメタデータ

[btree/node/record.md](../btree/node/record.md) に記載のとおり、B+Tree のレコードはヘッダー・キーフィールド・非キーフィールドの 3 領域で構成される

テーブルメタデータでは、各領域を以下のような内容で使用する (意味付けている)

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | テーブルメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId | テーブルの実データが格納されるディスクファイルの識別子 |
| 非キー | テーブル名やプライマリキーのカラム名などの情報 | - |

## インデックスメタデータ

同様に、インデックスメタデータでは、各領域を以下のような内容で使用する (意味付けている)

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | インデックスメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + インデックス名 | インデックスが属するテーブルの FileId とインデックス名を組み合わせたものをキーとすることで、テーブル内でインデックスを一意に識別できるようにしている |
| 非キー | インデックスの種類や、インデックスを構成するカラム名などの情報 | - |

_補足_

- セカンダリインデックスはユニーク・非ユニークの両方をサポートしている
- 複合インデックスはサポートしていない

## カラムメタデータ

同様に、カラムメタデータでは、各領域を以下のような内容で使用する (意味付けている)

| 領域 | 内容 | 説明 |
| --- | --- | --- |
| ヘッダー | - | カラムメタデータのレコードは削除されない想定なので、ヘッダーは使用しない |
| キー | FileId + カラム名 | カラムが属するテーブルの FileId とカラム名を組み合わせたものをキーとすることで、テーブル内でカラムを一意に識別できるようにしている |
| 非キー | カラムの順序位置やデータ型などの情報 | - |

_補足_

- 現状はカラムの型情報を保存しておらず、全て文字列型として扱っている
