# key-value ペア

## 参考文献

- [The physical structure of records in InnoDB](https://blog.jcole.us/2013/01/10/the-physical-structure-of-records-in-innodb/)

## 概要

- B+Tree のノードには、複数の key-value ペアが格納される
- key-value のペアはそれぞれエンコードされる
  - キーが複合キーの場合も考慮して (複合キーの場合も正しくソートされるように) Memcomparable format を使用してキーをエンコードしている ([参照](../about/memcomparable-format.md))

## クラスタ化インデックス

### ブランチノード

- key: 子ノードを分割する境界値が格納される
- value: key 未満の値を持つ子ノードのページ ID が格納される

※この辺りは [ブランチノード](./branch-node.md) をみた方がわかりやすい

### リーフノード

#### header

| フィールド | バイト数 | 説明 |
| --- | --- | --- |
| `delete_mark` | 1 | 削除マーク。削除された行はこのフラグが立つ。物理削除は後でパージによって行われる |

#### key

- テーブルのプライマリキーの値をエンコードしたものが格納される

#### value

以下の内容をエンコードしたものが格納される

| フィールド | バイト数 | 説明 |
| --- | --- | --- |
| `last_modified` | 8 | 自分を最後に変更したトランザクション ID |
| `roll_ptr` | 8 | 自分が更新される 1 世代前の行の内容 (undo ログ) へのポインタ (= ロールバックポインタ) |
| 非キーカラム値 | 可変 | プライマリキー以外のカラム値を結合したものをエンコードしたもの |

## セカンダリインデックス

### ブランチノード

クラスタ化インデックスと同様

### リーフノード

#### header

| フィールド | バイト数 | 説明 |
| --- | --- | --- |
| `delete_mark` | 1 | 削除マーク。削除された行はこのフラグが立つ。物理削除は後でパージによって行われる |

#### key

セカンダリインデックスを構成するカラム値をエンコードしたものが格納される

#### value

プライマリキーをエンコードしたものが格納される (`last_modified` や `roll_ptr` はセカンダリインデックスのペアには格納されない)
