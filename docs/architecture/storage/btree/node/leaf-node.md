# リーフノード

## 構造

| フィールド名 | バイト数 | 説明 |
| --- | --- | --- |
| ノードタイプヘッダー | 8 バイト | `"LEAF    "` が格納される |
| リーフノードヘッダー | 16 バイト | 前の PageId + 次の PageId が格納される |
| Slotted Page | 4072 バイト | ボディ部分 |

- リーフノードのデータ構造は Slotted Page になっているので、データの挿入などは [Slotted Page](./slotted-page.md) の仕様に従う
- Slotted Page のスロット数 = レコードの数

## リーフノードヘッダー

- サイズ: 16 byte
  - 前のリーフノード (=前のページ) へのポインタ: 8 byte
    - 前のリーフノードの PageId を格納する
  - 次のリーフノード (=次のページ) へのポインタ: 8 byte
    - 次のリーフノードの PageId を格納する

- 前のページへのポインタを読み取る際には、ページデータの 8 byte 目から 8 byte を読み取る
  - 同様に、次のページへのポインタは 16 byte 目から 8 byte を読み取る
- 書き込みの際も同様に、8 byte 目から 8 byte に前の PageId を、16 byte 目から 8 byte に次の PageId を書き込む

## レコードの挿入

- リーフノードが満杯でない場合
  - Slotted Page の仕様に従って、リーフノードのボディ部分にレコードを挿入する

- リーフノードが満杯の場合
  - 新しいリーフノードを作成し、自身のリーフノードのレコードのうち、半分を新しいリーフノードに移動させる
  - 新しいリーフノードに、挿入対象のレコードを挿入する

### 分割挿入の流れ

#### 1. 分割挿入先のリーフノードを作成する

- 新しいノードは古いノードの左側に配置される (古いノードより小さい値が格納される)

#### 2. 古いノードのレコードを新しいノードに移動しつつ、挿入対象のレコードを適切な位置に挿入する

- 挿入対象のレコードのキーと、分割前のリーフノード (古いノード) の先頭のキーを比較する
  - 古いノードの先頭レコードのキー < 挿入対象のキー: 古いノードの先頭レコードを分割先のリーフノード (新しいノード) に移動する
    - その過程で、新しいノードが半分埋まったら、挿入対象のレコードを古いノードに挿入して、ループ終了
  - 古いノードの先頭レコードのキー >= 挿入対象のキー: 挿入対象のレコードを新しいノードに挿入する
    - その後、新しいノードが半分埋まるまで、古いノードの先頭レコードを新しいノードに移動する

**_以下例_**

_例1: 挿入対象のキーが中間にある場合_

```txt
初期状態:
  - 古いノード (満杯): [10, 20, 30, 40, 50]  ← newRecord = 25 を挿入したい
  - 新しいノード (空):  []

ループ 1回目:
  - 新しいノードが半分埋まっているか? → No (空なので)
  - 古いノードの先頭(10) < 挿入対象(25) → 古いノードの先頭レコード(10)を新しいノードに移動
  - 結果
    - 古いノード: [20, 30, 40, 50]
    - 新しいノード: [10]

ループ 2回目:
  - 新しいノードが半分埋まっているか? → No (まだ半分未満)
  - 古いノードの先頭(20) < 挿入対象(25) → 古いノードの先頭レコード(20)を新しいノードに移動
  - 結果
    - 古いノード: [30, 40, 50]
    - 新しいノード: [10, 20]

ループ 3回目:
  - 新しいノードが半分埋まっているか? → No (まだ半分未満)
  - 古いノードの先頭(30) > 挿入対象(25) → 挿入対象(25)を新しいノードに挿入
  - その後、新しいノードが半分埋まるまで古いノードの先頭レコードを移動する
    - 新しいノード [10, 20, 25] → 半分埋まったので移動終了
  - 結果
    - 古いノード: [30, 40, 50]
    - 新しいノード: [10, 20, 25]
```

_例2: 挿入対象のキーが先頭より小さい場合_

```txt
初期状態:
  - 古いノード (満杯): [10, 20, 30, 40, 50]  ← newRecord = 5 を挿入したい
  - 新しいノード (空):  []

ループ 1回目:
  - 新しいノードが半分埋まっているか? → No (空なので)
  - 古いノードの先頭(10) >= 挿入対象(5) → 挿入対象(5)を新しいノードに挿入
  - その後、新しいノードが半分埋まるまで古いノードの先頭レコードを移動する
    - 1回目の移動: 古いノードの先頭(10)を新しいノードに移動
      - 古いノード: [20, 30, 40, 50], 新しいノード: [5, 10]
    - 2回目の移動: 古いノードの先頭(20)を新しいノードに移動
      - 古いノード: [30, 40, 50], 新しいノード: [5, 10, 20]
    - 新しいノードが半分埋まったので、移動終了
  - 結果
    - 古いノード: [30, 40, 50]
    - 新しいノード: [5, 10, 20]
```

## リーフノード内の特定のレコードの非キーを更新

- 更新前後でキーの値は変わらないと仮定する
- 非キー領域のリサイズを行う
  - リサイズの仕組みは [Slotted Page の仕様](./slotted-page.md#データのサイズ変更手順)に従う

## 兄弟ノードにレコードを転送する

- B+Tree におけるレコードの削除時には、兄弟ノードからレコードを転送するケースがある
  - 詳細: [B+Tree - リーフノードからの削除](../btree.md#リーフノードからの削除)
- その際に、転送元のノードが兄弟ノードにレコードを転送できるかどうかを判定する必要がある

### 転送可否の判定

自分自身のレコードを兄弟ノードに転送できるかどうかを判定する際には、転送対象のレコードを取り除いた後も自分が半分以上埋まっているかどうかで判定する

1. 指定された転送先に応じて、転送対象のレコードを決定する
   - 右の兄弟に転送する → 自分の末尾レコードを転送対象とする
   - 左の兄弟に転送する → 自分の先頭レコードを転送対象とする
2. 転送後の空き容量がリーフノード全体の容量の半分未満であれば、転送後も半分以上埋まっていると判断できるため、転送可能とする
