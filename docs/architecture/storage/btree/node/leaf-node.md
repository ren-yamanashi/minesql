# リーフノード

## 構造

| フィールド名 | バイト数 | 説明 |
| --- | --- | --- |
| ノードタイプヘッダー | 8 バイト | `"LEAF    "` が格納される |
| リーフノードヘッダー | 16 バイト | 前の PageId + 次の PageId が格納される |
| Slotted Page | 4072 バイト | ボディ部分 |

- リーフノードのデータ構造は Slotted Page になっているので、データの挿入などは [Slotted Page](./slotted-page.md) の仕様に従う
- Slotted Page のスロット数 = レコードの数

## リーフノードヘッダー

- サイズ: 16 byte
  - 前のリーフノード (=前のページ) へのポインタ: 8 byte
    - 前のリーフノードの PageId を格納する
  - 次のリーフノード (=次のページ) へのポインタ: 8 byte
    - 次のリーフノードの PageId を格納する

- 前のページへのポインタを読み取る際には、ページデータの 8 byte 目から 8 byte を読み取る
  - 同様に、次のページへのポインタは 16 byte 目から 8 byte を読み取る
- 書き込みの際も同様に、8 byte 目から 8 byte に前の PageId を、16 byte 目から 8 byte に次の PageId を書き込む

## レコードの挿入

- リーフノードが満杯でない場合
  - Slotted Page の仕様に従って、リーフノードのボディ部分にレコードを挿入する

- リーフノードが満杯の場合
  - 新しいリーフノードを作成し、自身のリーフノードのレコードのうち、半分を新しいリーフノードに移動させる
  - 新しいリーフノードに、挿入対象のレコードを挿入する

### 分割挿入の流れ

#### 1. 分割挿入先のリーフノードを作成する

- 新しいノードは古いノードの左側に配置される (古いノードより小さい値が格納される)

#### 2. 古いノードのレコードを新しいノードに移動しつつ、挿入対象のレコードを適切な位置に挿入する

- 挿入対象のレコードのキーと、分割前のリーフノード (古いノード) の先頭のキーを比較する
  - 古いノードの先頭レコードのキー < 挿入対象のキー: 古いノードの先頭レコードを分割先のリーフノード (新しいノード) に移動する
    - その過程で、新しいノードが半分埋まったら、挿入対象のレコードを古いノードに挿入して、ループ終了
  - 古いノードの先頭レコードのキー >= 挿入対象のキー: 挿入対象のレコードを新しいノードに挿入する
    - その後、新しいノードが半分埋まるまで、古いノードの先頭レコードを新しいノードに移動する

#### 3. 新しいリーフノードの先頭のキー値を親ノードに伝える

- リーフノードの分割が生じた際は、ブランチノードが持つべき境界値 (key) を増やす必要がある
- つまりブランチノードのレコードを増やす必要がある
- この時のブランチノードのレコードの key (=境界値) は、新しく作られたリーフノードの先頭のキーの値になる
- そのため、新しく作られたリーフノードの先頭のキー値を親ノード (ブランチノード) に伝える

**_以下例_**

_例1: 挿入対象のキーが中間にある場合_

```txt
初期状態:
  - 古いノード (満杯): [10, 20, 30, 40, 50]  ← newRecord = 25 を挿入したい
  - 新しいノード (空):  []

ループ 1回目:
  - 新しいノードが半分埋まっているか? → No (空なので)
  - 古いノードの先頭(10) < 挿入対象(25) → 古いノードの先頭レコード(10)を新しいノードに移動
  - 結果
    - 古いノード: [20, 30, 40, 50]
    - 新しいノード: [10]

ループ 2回目:
  - 新しいノードが半分埋まっているか? → No (まだ半分未満)
  - 古いノードの先頭(20) < 挿入対象(25) → 古いノードの先頭レコード(20)を新しいノードに移動
  - 結果
    - 古いノード: [30, 40, 50]
    - 新しいノード: [10, 20]

ループ 3回目:
  - 新しいノードが半分埋まっているか? → No (まだ半分未満)
  - 古いノードの先頭(30) > 挿入対象(25) → 挿入対象(25)を新しいノードに挿入
  - その後、新しいノードが半分埋まるまで古いノードの先頭レコードを移動する
    - 新しいノード [10, 20, 25] → 半分埋まったので移動終了
  - 結果
    - 古いノード: [30, 40, 50]
    - 新しいノード: [10, 20, 25]
```

_例2: 挿入対象のキーが先頭より小さい場合_

```txt
初期状態:
  - 古いノード (満杯): [10, 20, 30, 40, 50]  ← newRecord = 5 を挿入したい
  - 新しいノード (空):  []

ループ 1回目:
  - 新しいノードが半分埋まっているか? → No (空なので)
  - 古いノードの先頭(10) >= 挿入対象(5) → 挿入対象(5)を新しいノードに挿入
  - その後、新しいノードが半分埋まるまで古いノードの先頭レコードを移動する
    - 1回目の移動: 古いノードの先頭(10)を新しいノードに移動
      - 古いノード: [20, 30, 40, 50], 新しいノード: [5, 10]
    - 2回目の移動: 古いノードの先頭(20)を新しいノードに移動
      - 古いノード: [30, 40, 50], 新しいノード: [5, 10, 20]
    - 新しいノードが半分埋まったので、移動終了
  - 結果
    - 古いノード: [30, 40, 50]
    - 新しいノード: [5, 10, 20]
```

## レコードの削除

リーフノードからの削除は以下の手順で行う

1. 二分探索により、削除すべきレコードを特定
2. レコードを削除
3. ノードの空き容量が閾値 (ノードの最大容量の半分) を下回る場合、以下のいずれかを行う
   - 兄弟ノードのレコードを自分のノードに移動
     - 基本的には右の (つまり自分より大きいキーを持つ) 兄弟ノードから転送が行われるが、右端のノードの場合は左の兄弟ノードから転送が行われる
   - 兄弟ノードとマージする
     - 兄弟のノードから転送が行われると、兄弟のノードの空き容量が閾値を下回る場合、兄弟ノードとマージする
4. ブランチノードのキーを更新する
   - 3 の処理でマージを行った場合はキーの更新が必要であるが、ノードが移動した場合も、兄弟との境界線が変わるため、ブランチノードのキーの更新が必要になる

### 例1. ノードの再分割が発生する場合

- 初期のB+Tree の構造:

    ```txt
                [Root: Branch Node]
                Records: [(key="fish", pageId=10), (key="monkey", pageId=20)]
                RightChild: pageId=30
                       /            |              \
                  (<fish)         (fish~monkey)    (>=monkey)
                  PageId=10       PageId=20       PageId=30
                    /                |                \
     [Leaf: apple,cat,dog]  [Leaf: fish,grape,lion]  [Leaf: monkey,tiger,zebra]
    ```

- 仮定
  - `key = "cat"` を削除する
  - 削除後にリーフノードの空き容量が閾値を下回る
- 流れ
  1. 二分探索により、削除すべきレコードを特定 (割愛)
  2. レコードを削除 (割愛)
  3. ノードの空き容量が閾値を下回るため、右の兄弟ノード (PageId=20) からレコードを移動
     - 右の兄弟ノード (PageId=20) から最小のレコード (key="fish") を移動
     - もともとは [apple, cat, dog] の順であったが、cat を削除して fish を移動したため、[apple, dog, fish] の順になる
     - 移動後のリーフノード (PageId=10): [apple, dog, fish]
     - 移動後の右の兄弟ノード (PageId=20): [grape, lion]
  4. ブランチノードのキーを更新
     - 移動前の境界線は "dog" と "fish" の間であったが、移動後の境界線は "fish" と "grape" の間になるため、ブランチノードのキーを "grape" に更新

### 例2. ノードのマージが発生する場合 (左端と真ん中のマージ)

- 初期のB+Tree の構造:

    ```txt
                [Root: Branch Node]
                Records: [(key="fish", pageId=10), (key="monkey", pageId=20)]
                RightChild: pageId=30
                       /            |              \
                  (<fish)         (fish~monkey)    (>=monkey)
                  PageId=10       PageId=20       PageId=30
                    /                |                \
     [Leaf: apple,cat]  [Leaf: fish,grape]  [Leaf: monkey,tiger,zebra]
    ```

- 仮定
  - `key = "cat"` を削除する
  - 削除後にリーフノードの空き容量が閾値を下回る
  - また右の兄弟ノード (PageId=20) からレコードを移動しても、右の兄弟ノードの空き容量が閾値を下回る

- 流れ
  1. 二分探索により、削除すべきレコードを特定 (割愛)
  2. レコードを削除 (割愛)
  3. ノードの空き容量が閾値を下回るため、右の兄弟ノード (PageId=20) からレコードを移動したいが、そうすると右の兄弟ノードの空き容量も閾値を下回るため、右の兄弟ノードとマージする
     - マージ後のリーフノード (PageId=10): [apple, fish, grape]
     - マージ後の右の兄弟ノード (PageId=20) は不要になるため、削除する
  4. ブランチノードのキーを更新
     - マージ前の境界線は "cat" と "fish" の間であったが、マージ後の境界線は "grape" と "monkey" の間になるため、ブランチノードのキーを "monkey" に更新
     - ブランチノードに紐づくリーフノードの数が K から K-1 になるため、ブランチノードのレコードも削除する

- 削除後のB+Tree の構造:

    ```txt
                [Root: Branch Node]
                Records: [(key="monkey", pageId=10)]
                RightChild: pageId=30
                       /            |          
                  (<monkey)       (>=monkey)
                  PageId=10       PageId=30
                    /                |
     [Leaf: apple,fish,grape]  [Leaf: monkey,tiger,zebra]
    ```

### 例3. ノードのマージが発生する場合 (真ん中と右端のマージ)

- 初期のB+Tree の構造:

    ```txt
                [Root: Branch Node]
                Records: [(key="fish", pageId=10), (key="monkey", pageId=20)]
                RightChild: pageId=30
                       /            |              \
                  (<fish)         (fish~monkey)    (>=monkey)
                  PageId=10       PageId=20       PageId=30
                    /                |                \
     [Leaf: apple,cat]  [Leaf: fish,grape]  [Leaf: monkey,tiger]
    ```

- 仮定
  - `key = "monkey"` を削除する
  - 削除後にリーフノードの空き容量が閾値を下回る
  - また左の兄弟ノード (PageId=20) からレコードを移動しても、左の兄弟ノードの空き容量が閾値を下回る

- 流れ
  1. 二分探索により、削除すべきレコードを特定 (割愛)
  2. レコードを削除 (割愛)
  3. ノードの空き容量が閾値を下回るため、左の兄弟ノードからレコードを移動したいが、そうすると左の兄弟ノードの空き容量も閾値を下回るため、左の兄弟ノードとマージする
     - マージ後のリーフノード (PageId=20): [fish,grape,tiger]
     - 自分のノード（PageID=30）は空になり不要になるため、削除する
  4. ブランチノードのキーを更新
     - マージ前の境界線は "grape" と "monkey" の間であったが、マージ後の境界線は "cat" と "fish" の間になるため、ブランチノードのキーを "fish" に更新
     - ブランチノードに紐づくリーフノードの数が K から K-1 になるため、ブランチノードのレコードも削除する
     - また PageId=20 がRightChild に昇格する

- 削除後のB+Tree の構造:

    ```txt
                [Root: Branch Node]
                Records: [(key="fish", pageId=10)]
                RightChild: pageId=20
                       /            |          
                  (<fish)       (>=fish)
                  PageId=10       PageId=20
                    /                |
     [Leaf: apple,cat]  [Leaf: fish,grape,tiger]
    ```

※ ポイントとして、マージの際は常に「左のノードに右のノードをマージする」というルールにしている (要するに、マージの際には常に左のノードが残る)\
このようにしているのは、「右のノードに左のノードをマージする」と比較して高速であるため\
(append だとシフト操作が不要なので prepend よりも高速であるため)

#### 転送不可かつマージ不可のケース

- 通常、転送すると転送元の空き容量が閾値を下回ると (容量の半分以下になると) 転送不可と判断される
- またその場合は転送ではなく、兄弟ノードとのマージが選択される
- ただし以下のような場合では、転送不可と判断されても 2 つのノードの合計データ量が 1 ノードの容量を超えてしまい、マージできないケースがある
  - アンダーフローしたノード A の空き容量: `≒ capacity/2`
  - 兄弟ノード B の使用量: `≒ capacity/2 + a` (α はレコードのサイズ)
  - この場合、ノード B の レコード a を転送すると、転送後のノード B は閾値を下回る
  - ただしマージすると、マージ後のノードの容量は `capacity + a` となる
- このような場合は、マージを注視してアンダーフロー (一時的にノードの空き容量が閾値を下回ること) をそのまま許容する

## リーフノード内の特定のレコードの非キーを更新

- 更新前後でキーの値は変わらないと仮定する
- 非キー領域のリサイズを行う
  - リサイズの仕組みは [Slotted Page の仕様](./slotted-page.md#データのサイズ変更手順)に従う

## 兄弟ノードにレコードを転送する

- B+Tree におけるレコードの削除時には、兄弟ノードからレコードを転送するケースがある
  - 詳細: [B+Tree - リーフノードからの削除](../btree.md#リーフノードからの削除)
- その際に、転送元のノードが兄弟ノードにレコードを転送できるかどうかを判定する必要がある

### 転送可否の判定

自分自身のレコードを兄弟ノードに転送できるかどうかを判定する際には、転送対象のレコードを取り除いた後も自分が半分以上埋まっているかどうかで判定する

1. 指定された転送先に応じて、転送対象のレコードを決定する
   - 右の兄弟に転送する → 自分の末尾レコードを転送対象とする
   - 左の兄弟に転送する → 自分の先頭レコードを転送対象とする
2. 転送後の空き容量がリーフノード全体の容量の半分未満であれば、転送後も半分以上埋まっていると判断できるため、転送可能とする
