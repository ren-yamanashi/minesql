# Node

## ノードの種類

- [リーフノード](./leaf-node.md)
- [ブランチノード](./branch-node.md)

## ノード構造

- すべてのノードは、先頭に 8 バイトのノードタイプヘッダーを持つ
  - ノードタイプヘッダー: 8 バイト
    - `"LEAF    "` (リーフノード) または `"BRANCH  "` (ブランチノード)
    - ページデータの先頭 8 バイトを読み取ることで、ノード型を判定できる

## レコード

- ノードには複数のレコードが格納される
- レコードのバイト列は以下のように構成される

  ```txt
  [headerSize (2B)][keySize (2B)][header (可変長)][key (可変長)][nonKey (可変長)]
  ```

  - 先頭 4 バイトのサイズフィールドにより、可変長の 3 領域を分離できる
  - nonKey のサイズは `全体の長さ - 4 - headerSize - keySize` で算出する

## 指定したキーを持つレコードを見つける方法

### 前提

- ノード内には複数のレコードが格納されている
- ノード内のレコードはキーの昇順でソートされている
- ノードはレコード数を保持している (自分のノードの中にいくつのレコードが存在するかの情報)
- 1 ノード = 1 ページ = 4096 バイト である
  - 特定の一つのページに複数の B+Tree ノードのデータが格納されることはない
  - MySQL の InnoDB においてもおそらくその設計 (1ページ=1ノード) になっている
    - 参考: https://planetscale.com/blog/btrees-and-database-indexes

### 探索の流れ

1. ノード内のレコード数を取得する
2. `0 ~ レコード数` の範囲で、ノード内のレコードを二分探索する
   - レコード (二分探索でいうところの mid に位置するレコード) のキーと、検索したいキーを比較する
   - 比較の結果
     - 一致: 目的のレコードが見つかったと判断する
     - レコードのキー < 検索したいキー: 探索範囲を右半分に絞る
     - レコードのキー > 検索したいキー: 探索範囲を左半分に絞る
3. レコードが見つかるか、探索範囲がなくなるまで `2` の流れを繰り返す
   - 最後まで目的のレコードが見つからなければ、ノード内にそのキーは存在しないと判断する
     - 最後まで: left >= right になるまで (left と right が徐々に近づいていき、最終的に同じ位置を指すまたは交差するまで)
